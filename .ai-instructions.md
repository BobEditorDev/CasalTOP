This file contains specific instructions for AI coding agents (Claude, GPT, Gemini, SWE-1.6, etc.) working on the CasalTOP project.

## 🎯 Primary Directives

1. **READ AGENTS.md FIRST** - Always read the full AGENTS.md file before making changes
2. **Follow Domain Structure** - Never deviate from the established domain structure
3. **Use Established Patterns** - Follow the existing code patterns exactly

## 🚫 Forbidden Actions

- ❌ NEVER create files outside the established structure
- ❌ NEVER mix responsibilities between layers (actions, services, repositories)
- ❌ NEVER use `any` type - always use proper TypeScript types
- ❌ NEVER skip validation - always use Zod schemas
- ❌ NEVER access Prisma directly outside repositories
- ❌ NEVER put business logic in components
- ❌ NEVER skip 'use server' in actions files
- ❌ NEVER forget revalidatePath after mutations

## ✅ Required Actions

- ✅ ALWAYS follow the domain structure pattern
- ✅ ALWAYS use Server Actions for write operations
- ✅ ALWAYS validate with Zod before processing
- ✅ ALWAYS use TypeScript strict mode
- ✅ ALWAYS use proper error handling
- ✅ ALWAYS update index.ts exports when adding new functions
- ✅ ALWAYS run architecture validation before committing

## 📁 File Creation Patterns

### When creating a NEW domain:
```
1. Create folder structure: actions/, components/, repositories/, services/, types/, validations/, utils/
2. Create types/index.ts with interfaces
3. Create validations/index.ts with Zod schemas
4. Create repositories/index.ts with Prisma access
5. Create services/index.ts with business logic
6. Create actions/index.ts with Server Actions
7. Create components/ for UI if needed
8. Create index.ts with public exports
9. Create route in app/
10. Update sidebar if needed
```

### When adding to EXISTING domain:
```
1. Add type to types/index.ts
2. Add validation to validations/index.ts
3. Update repository if needed
4. Update service if needed
5. Update action if needed
6. Update component if needed
7. Update index.ts exports
```

## 🔍 Code Patterns to Follow

### Server Action Pattern:
```typescript
'use server'

import { service } from '../services'
import { schema } from '../validations'
import { revalidatePath } from 'next/cache'

export async function createAction(formData: FormData) {
  const data = { /* extract from formData */ }
  const validated = schema.parse(data)
  await service.create(validated)
  revalidatePath('/route')
}
```

### Service Pattern:
```typescript
import { repository } from '../repositories'

export class Service {
  async create(data: InputType) {
    return repository.create(data)
  }
}
```

### Repository Pattern:
```typescript
import { prisma } from '@/lib/prisma'

export class Repository {
  async findAll() {
    return prisma.entity.findMany()
  }
}
```

## 🧪 Testing Your Changes

Before considering work complete:

1. Run architecture validation: `npm run validate:architecture`
2. Run TypeScript check: `npm run typecheck`
3. Run linter: `npm run lint`
4. Test the specific feature manually if possible
5. Check for any console errors

## 🚨 Common Mistakes to Avoid

1. **Importing repositories in components** - Use services/actions instead
2. **Forgetting 'use server'** - Server Actions must have this directive
3. **Skipping validation** - Always validate with Zod
4. **Direct Prisma access** - Only repositories should access Prisma
5. **Missing revalidatePath** - Always revalidate after mutations
6. **Breaking domain structure** - Follow the established pattern
7. **Using any type** - Always use proper TypeScript types
